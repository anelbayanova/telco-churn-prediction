# Проект: Прогнозирование оттока клиентов (Telco Churn Prediction)

### Контекст и бизнес-задача
Цель проекта — разработать модель, предсказывающую отток клиентов телеком-оператора. По условиям бизнеса, **стоимость пропуска уходящего клиента (False Negative) примерно в 5 раз выше, чем стоимость звонка лояльному клиенту (False Positive)**. 

В связи с этим основной фокус при разработке был сделан на метрику **Recall** (максимизация охвата уходящих) при удержании контролируемого уровня **Precision** (точности), чтобы маркетинг не тратил бюджет на обзвон всей базы.

---

## 1. Архитектура решения и защита от Data Leakage

Вся предобработка данных и генерация признаков инкапсулированы внутри единого scikit-learn пайплайна с использованием `ColumnTransformer`. Это гарантирует 100% идентичность обработки на этапе обучения и инференса, а также полностью исключает утечку данных (data leakage), так как трансформеры обучаются строго на тренировочных фолдах.

### Обоснование и генерация новых фич (внутри пайплайна):
Обычный Pandas-препроцессинг до деления выборки не использовался. В кастомных трансформерах `FixTotalCharges` и `AddFeatures` реализованы:
1. **`TotalCharges` (Капкан с типами данных):** Колонка содержала скрытые пробелы `" "`, из-за чего считывалась как текст. Мы принудительно перевели её в числовой формат через `pd.to_numeric(..., errors='coerce')`, а образующиеся `NaN` заполнили медианой с помощью `SimpleImputer` уже на этапе числовой предобработки пайплайна.
2. **`tenure_bucket`:** На этапе EDA было выявлено, что пик оттока приходится на первые 12 месяцев. Срок жизни клиента (`tenure`) был разбит на бизнес-корзины (`0-12`, `13-24`, `25-48`, `49+`).
3. **`n_services` (Капкан со стримингами):** Подсчет количества активных услуг. Простая проверка `== 'Yes'` здесь не работает, так как `InternetService` принимает текстовые значения (`DSL`, `Fiber optic`). Логика реализована через инвертированный поиск неактивных статусов (`No`, `No internet service`), что позволило корректно учесть все типы подписок.
4. **`charges_per_tenure`:** Отношение общих затрат к сроку жизни клиента для выявления аномалий в тарификации.

## 2. Валидация и сравнение моделей

Данные разделены на Train/Test в пропорции 80/20 с применением стратификации (`stratify=y`) для сохранения баланса классов (~26.5% оттока). Тестовый сет был отложен и использован **строго один раз** в самом конце.

Поскольку базовая точность (Accuracy) при константном предсказании составляет 73.5%, эта метрика была отброшена как неинформативная. Модели сравнивались на 5-fold Stratified Cross-Validation.

### Результаты кросс-валидации:

| Модель | Recall | Precision | F1 | ROC-AUC |
| :--- | :---: | :---: | :---: | :---: |
| **DummyMostFrequent** | 0.000 | 0.000 | 0.000 | 0.500 |
| **DummyStratified** | 0.278 | 0.275 | 0.276 | 0.507 |
| **LogisticRegression (balanced)** | **0.797** | 0.517 | 0.627 | 0.846 |
| **RandomForest** | 0.465 | 0.639 | 0.538 | 0.823 |
| **GradientBoosting** | 0.520 | **0.660** | 0.582 | **0.847** |

*Примечание: Чтобы избежать предупреждений `UndefinedMetricWarning` при делении на ноль на константном бейслайне, метрики были обернуты в `make_scorer(..., zero_division=0)`.*

### Тюнинг:
Для финального решения был выбран **GradientBoostingClassifier**, как наиболее стабильный алгоритм с наилучшим разделением классов (ROC-AUC = 0.847). Параметры оптимизировались через `GridSearchCV` по метрике `average_precision` (PR-AUC). 
* *Лучшие параметры:* `learning_rate: 0.05`, `max_depth: 3`, `n_estimators: 100`.

---

## 3. Подбор бизнес-порога

Дефолтный порог `0.5` не отвечает требованиям бизнеса по максимизации Recall. Оптимальный порог подбирался **строго на out-of-fold предсказаниях тренировочной выборки** (через `cross_val_predict`). 

Был зафиксирован порог **`0.2642`**, который гарантирует достижение $Recall \ge 0.80$ на этапе обучения при максимально доступной точности.

---

## 4. Финальные результаты на тест-сете

Пайплайн и зафиксированный порог были применены к отложенному тест-сету всего один раз. Модель показала высокую обобщающую способность, выполнив все требования ТЗ с запасом:

* **Recall (churn):** `0.7968` *(Цель: >= 0.75)* — модель находит 4 из 5 уходящих клиентов.
* **Precision (churn):** `0.5183` *(Цель: >= 0.45)* — каждый второй звонок маркетолога попадает точно в цель.
* **ROC-AUC:** `0.8441`

### Матрица ошибок (Confusion Matrix):
* **True Stay:** 758 | **False Churn (Ложный вызов):** 277
* **False Stay (Пропущенный отток):** 76 | **True Churn (Пойманный отток):** 298

Бизнес-интерпретация результатов и подробный финансовый отчет для стейкхолдеров находятся в файле `results/results.md`.

## 5. Инструкция по запуску проекта

Пайплайн полностью сериализован в `results/churn_pipeline.pkl`. Все кастомные трансформеры вынесены в `scripts/preprocessing.py`, что позволяет разворачивать модель на инференс без дублирования кода.

```bash
# 1. Клонируем репозиторий и переходим в папку проекта
git clone [https://github.com/anelbayanova/telco-churn-prediction.git]
cd telco-churn-prediction

# 2. Создаем и активируем виртуальное окружение (venv)
python3 -m venv venv

# Для macOS / Linux:
source venv/bin/activate
# Для Windows (Command Prompt):
# venv\Scripts\activate.bat

# 3. Устанавливаем зависимости
pip install --upgrade pip
pip install -r requirements.txt

# 4. Запуск полного цикла обучения и валидации
python3 scripts/train.py

# 5. Запуск инференса (предсказания для новых данных)
python3 scripts/predict.py путь_к_файлу.csv
```
